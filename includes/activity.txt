Meme-tastic data science with OpenAI in the Tidyverse
A practical, hands-on activity for grad students to learn how to call OpenAI’s API from R, manage responses with the Tidyverse, and analyze the results like a data scientist.

Overview
This activity guides you through three interconnected parts:
- Part 1: Generate meme captions about data science topics using OpenAI's chat completion API, and organize results with tidyverse tools.
- Part 2: Prompt the API to generate R code for a simple data-science task, then attempt to sandbox-test and run the produced code.
- Part 3: Analyze and visualize the API outputs (captions, token usage, and cost) with tidyverse, and reflect on prompt engineering choices.

Learning objectives
- Learn how to set up and call the OpenAI chat API from R (using the Tidyverse for data handling).
- Practice prompt engineering: crafting system/user prompts to elicit meme captions and code snippets.
- Learn to parse and handle API responses, including usage tokens and response content.
- Build a reproducible tidyverse workflow to analyze results, visualize trends, and estimate API costs.
- Safely test generated code in a sandboxed environment and reflect on reliability and safety.

Prerequisites
- R (4.x recommended) and a recent RStudio session.
- OpenAI API key (set as an environment variable OPENAI_API_KEY; do not hard-code keys).
- Familiarity with:
  - Tidyverse (dplyr, purrr, tidyr, tibble, ggplot2, readr)
  - Basic HTTP requests in R (httr) and JSON parsing (jsonlite)
- Optional: basic familiarity with prompt engineering concepts.

Part 1. Meme caption generation with OpenAI and tidyverse

Goal: Use OpenAI’s chat completion endpoint to generate witty meme captions about data science topics, then organize and inspect the results with tidyverse.

What you’ll do
- Create a small list of data-science topics you want memes about.
- For each topic, prompt the API to generate a concise meme caption in a meme-template style (e.g., Distracted Boyfriend, Two Buttons).
- Record the caption along with token usage and an estimated cost.
- Visualize and summarize the captions and costs.

Key functions to implement (copy-paste into your R script)

- API call helper (chat/completions)
  - This function sends a chat-based prompt and returns the parsed response and usage.

  call_openai_chat <- function(model = "gpt-4", messages, max_tokens = 200, temperature = 0.7) {
    api_key <- Sys.getenv("OPENAI_API_KEY")
    if (api_key == "") stop("OPENAI_API_KEY environment variable is not set.")
    req <- httr::POST(
      url = "https://api.openai.com/v1/chat/completions",
      httr::add_headers(Authorization = paste0("Bearer ", api_key)),
      httr::content_type_json(),
      httr::body = jsonlite::toJSON(
        list(model = model,
             messages = messages,
             max_tokens = max_tokens,
             temperature = temperature),
        auto_unbox = TRUE
      )
    )
    httr::stop_for_status(req)
    res <- httr::content(req, as = "parsed", simplifyVector = TRUE)
    res
  }

- Extract caption and usage
  extract_caption <- function(res) {
    # OpenAI chat responses place content at choices[[i]]$message$content
    if (!is.null(res$choices) && length(res$choices) > 0) {
      content <- res$choices[[1]]$message$content
      return(content)
    }
    NA_character_
  }

  extract_usage <- function(res) {
    if (!is.null(res$usage)) res$usage
    else NULL
  }

- Build a single-caption generator (per topic)
  generate_meme_caption <- function(topic, template = "Distracted Boyfriend", model = "gpt-4", max_tokens = 200, temperature = 0.7) {
    prompt <- paste0(
      "Create a witty, concise meme caption for a data science topic: '", topic, "'. ",
      "Use a meme-template style like ", template, ". Keep it under 20 words and accessible to a data-science audience."
    )
    messages <- list(
      list(role = "system", content = "You are a witty data science meme generator."),
      list(role = "user", content = prompt)
    )
    resp <- call_openai_chat(model = model, messages = messages, max_tokens = max_tokens, temperature = temperature)
    caption <- extract_caption(resp)
    usage <- extract_usage(resp)
    list(
      caption = caption,
      usage = usage,
      model = model,
      temperature = temperature,
      template = template
    )
  }

- Cost estimation helper (rough, per-token pricing)
  estimate_cost <- function(usage, model = "gpt-4") {
    if (is.null(usage)) return(NA_real_)
    # Costs (as of typical pricing; adjust to your plan)
    # GPT-4 chat: ~$0.03 per 1k prompt tokens; $0.06 per 1k completion tokens
    if (model %in% c("gpt-4", "gpt-4-0613", "gpt-4-8k")) {
      prompt_tokens <- if (!is.null(usage$prompt_tokens)) usage$prompt_tokens else 0
      completion_tokens <- if (!is.null(usage$completion_tokens)) usage$completion_tokens else 0
      total_cost <- (prompt_tokens / 1000) * 0.03 + (completion_tokens / 1000) * 0.06
      return(total_cost)
    }
    # GPT-3.5-turbo (cheaper, example)
    if (model == "gpt-3.5-turbo") {
      total_tokens <- if (!is.null(usage$total_tokens)) usage$total_tokens else 0
      return((total_tokens / 1000) * 0.0015)
    }
    NA_real_
  }

- Run a batch of topics and collect results
  topics <- tibble(
    topic = c(
      "Overfitting in small datasets",
      "P-hacking and p-values misuse",
      "Data wrangling with dplyr",
      "Reproducible research practices",
      "Model interpretability and SHAP"
    ),
    template = c("Distracted Boyfriend", "Two Buttons", "Mock Brand Meme Template", "Success Kid", "Drake Hotline Bling")
  )

  memes <- topics %>%
    mutate(
      # For each row, call the API and store results as a list column
      api_out = pmap(list(topic, template), ~ generate_meme_caption(..1, template = ..2)),
      caption = map(api_out, "caption"),
      usage = map(api_out, "usage"),
      model = map(api_out, "model"),
      temperature = map(api_out, "temperature"),
      template_used = map(api_out, "template")
    )

  # Flatten the data into a tidy, analyzable table
  memes_tbl <- memes %>%
    select(topic, template, caption, usage, model, temperature, template_used) %>%
    mutate(
      prompt_tokens = map_int(usage, ~ if (!is.null(.x$prompt_tokens)) .x$prompt_tokens else NA_integer_),
      completion_tokens = map_int(usage, ~ if (!is.null(.x$completion_tokens)) .x$completion_tokens else NA_integer_),
      total_tokens = map_int(usage, ~ if (!is.null(.x$total_tokens)) .x$total_tokens else NA_integer_),
      cost_usd = map_dbl(usage, ~ estimate_cost(.x, model = ifelse(is.null(.x$model), "gpt-4", .x$model)))
    ) %>%
    select(topic, caption, template, template_used, prompt_tokens, completion_tokens, total_tokens, cost_usd)

- Quick checks and visualizations
  - Caption length
  memes_tbl <- memes_tbl %>% mutate(words = str_count(caption, "\\w+"), chars = str_length(caption))

  # Plot caption length by topic
  ggplot(memes_tbl, aes(x = reorder(topic, words), y = words)) +
    geom_col(fill = "steelblue") +
    coord_flip() +
    labs(title = "Meme Caption Word Count by Topic", x = "Topic", y = "Word count")

  # Cost distribution
  ggplot(memes_tbl, aes(x = topic, y = cost_usd)) +
    geom_col(fill = "tomato") +
    coord_flip() +
    labs(title = "Estimated OpenAI API Cost by Topic", x = "Topic", y = "Estimated cost (USD)")

What you should observe
- A small tibble with topic, caption, template, token usage, and estimated cost per caption.
- Basic visualizations showing which topics produced longer captions and/or higher estimated costs.

Notes and caveats
- OpenAI policies: Use safe, policy-conforming prompts (avoid disallowed content). You are responsible for prompt content.
- API usage and costs vary by model and token counts. The cost estimates are rough guides, not precise invoices.
- Network calls can fail; wrap API calls with tryCatch in production use for robust pipelines.

Part 2. Prompting for code: generate and test R code

Goal: Have the model produce R code for a data-science task, then safely test the generated code in a sandboxed environment.

What you’ll do
- Prompt the model to output a self-contained R function or snippet that performs a defined task (e.g., fit a model and return a tidy summary).
- Capture the generated code, then attempt to run it in a sandbox environment, catching errors instead of crashing.
- Evaluate the output (does the function return reasonable results? Is the code self-contained and readable?).

Code prompts and test harness (one approach)

- Code prompt helper
  generate_code_for_task <- function(task_description, model = "gpt-4", max_tokens = 400, temperature = 0.3) {
    prompt <- paste0(
      "Provide a complete, copy-pasteable R  code snippet that ", task_description,
      ". The snippet should define a function named 'task_result' that takes a data.frame as input and returns a tidy tibble with summary results. Include comments and avoid external dependencies beyond base/tidyverse."
    )
    messages <- list(
      list(role = "system", content = "You are an expert R data scientist who writes clean, reproducible code."),
      list(role = "user", content = prompt)
    )
    resp <- call_openai_chat(model = model, messages = messages, max_tokens = max_tokens, temperature = temperature)
    code <- extract_caption(resp)  # In this API usage, we’re treating the code as the "caption" of the result
    list(code = code, usage = extract_usage(resp))
  }

- Sandbox runner (very safe approach)
  run_generated_code <- function(code_string) {
    if (is.na(code_string) || code_string == "") return(list(ok = FALSE, error = "Empty code"))
    env <- new.env(parent = emptyenv())
    # Try to evaluate the code in a sandbox
    result <- tryCatch(
      {
        eval(parse(text = code_string), envir = env)
        # If the code defines a function named 'task_result', try running it on a sample dataset
        if (exists("task_result", envir = env, inherits = FALSE)) {
          # Example usage: run on mtcars with a simple return
          res <- with(mtcars, task_result(mtcars))
          list(ok = TRUE, result = res)
        } else {
          list(ok = FALSE, error = "Code did not define expected function 'task_result'")
        }
      },
      error = function(e) list(ok = FALSE, error = e$message)
    )
    result
  }

- Example workflow
  task_desc <- "return a tibble with coef(3) and R-squared for a simple linear model mpg ~ wt using the input data.frame"

  code_resp <- generate_code_for_task(task_desc)
  code_string <- code_resp$code
  test <- run_generated_code(code_string)

  # Inspect test$result if ok, or inspect test$error if not ok.

Safety and helpful tips
- Do not blindly execute code received from the API in a live environment. Use a dedicated sandbox (new R session, restricted environment).
- The API’s output may be imperfect or partial; expect to edit and refine the code manually.
- Encourage participants to comment and add error handling to the generated code.

Part 3. Analyzing and interpreting API outputs with tidyverse

Goal: Use the results from Part 1 (captions and token usage) to perform a small analysis, produce visuals, and reflect on the quality and reliability of the outputs.

What you’ll do
- Compare caption lengths and topics.
- Explore token usage and estimated cost per caption.
- Optionally perform a quick sentiment/quality check on the captions using tidytext (e.g., basic sentiment scores) to see if some topics yield more positive captions.
- Save the results for reproducibility.

Discussion prompts for students
- Which topics yielded the shortest or longest captions, and how did that relate to perceived meme effectiveness?
- How stable are token counts and estimated costs across prompts for similar topics?
- How could prompt templates influence caption quality or clarity? What changes would you test next?

Optional extensions
- Cross-model comparison: run the same prompts with gpt-3.5-turbo and compare caption quality and cost.
- Multi-turn prompts: incorporate a system message instructing the model to adopt a specific meme template across batches and compare results.
- Additional metrics: compute readability scores (e.g., Flesch-Kurti Readability) or use simple lexical diversity metrics to compare captions.

Deliverables and assessment
- A reproducible R script or R Markdown notebook with:
  - API call logic and error handling
  - A small dataset of topics and meme templates
  - A tidyverse-based pipeline to generate captions, collect usage data, and estimate costs
  - Visualizations summarizing caption length and cost by topic
  - A brief reflection on prompt design choices and reliability
- Optional: a short module showing generated code for a task and a sandboxed test result

Tips for a successful session
- Start with a quick demo: show a sample topic and a generated caption, then walk through the resulting tibble and visualizations.
- Emphasize reproducibility: lock package versions with renv if you plan to share notebooks across cohorts.
- Remind students to manage API usage mindfully; start with low max_tokens and moderate temperature to keep responses concise and cost-effective.
- Encourage critical evaluation: not every caption will be perfect; use the activity to practice iterative prompt design.

Safety and ethics notes
- Respect API terms and usage policies.
- Avoid prompts that produce disallowed content or biased or harmful material. Review prompts with a focus on safe, inclusive, and professional communication.
- Do not expose or share API keys; use environment variables and local, reproducible environments.
