Meme-tastic Data Science with OpenAI API and Tidyverse (Graduate-level activity)

Overview
- Goal: Learn to make OpenAI API calls from R, handle JSON responses, and analyze model-generated outputs within the Tidyverse workflow.
- Theme: Build data-driven memes from a real dataset, compare prompt variations, and quantify how prompts shape model outputs.
- Deliverables: A tidy data frame of meme captions, basic qualitative/quantitative analyses, and a short reflective write-up on prompt design and limitations.

Learning objectives
- Practice making REST API calls to OpenAI from R (using tidyverse-friendly workflows).
- Parse and handle JSON responses, extract the text you need, and store results in tibbles.
- Design and compare multiple prompts (prompt engineering) to elicit diverse meme captions.
- Analyze generated text with basic NLP/text analytics (length, word count, sentiment) and visualize results.
- Reflect on model limitations, biases, rate limits, and cost implications.

Prerequisites
- R and RStudio (or any R environment)
- Basic familiarity with the Tidyverse (dplyr, purrr, tibble, tibble, ggplot2)
- An OpenAI API key (set as an environment variable OPENAI_API_KEY)
- R packages: httr, jsonlite, dplyr, purrr, tibble, stringr, readr, ggplot2, tidyr, tidytext, glue, forcats (install via install.packages)

Data and theme
- Dataset: Use the built-in mtcars dataset, augmented with car names.
- Promise: For each car, generate a meme caption based on its features (mpg, hp, wt, qsec) using OpenAI. Then analyze caption properties.

Activity outline (4 phases)
Phase 1 — setup and data prep (15–25 min)
- Step 1: Load libraries and prepare the data
  - Create a tibble from mtcars with a name column (car model names), and select relevant features.
  - Example data fields: name, mpg, hp, wt, qsec.
- Step 2: Define prompt strategies (two prompts to compare)
  - Prompt A: Meme caption in a playful style (e.g., Doge or Drake meme vibe), based on the row data.
  - Prompt B: Data-insight caption—briefly explains a data insight inferred from the row, in a witty meme style.
- Step 3: Define error handling and rate-limiting basics (basic retry/backoff)

Phase 2 — API calls and response handling (30–45 min)
- Step 4: Implement an API call function in R
  - Use httr to POST to https://api.openai.com/v1/chat/completions
  - Build a messages list with a system prompt and a user prompt that includes the row’s data
  - Use model gpt-3.5-turbo (or your preferred model) with a reasonable max_tokens, and a modest temperature
  - Extract the content of choices[[1]]$message$content
  - Example structure (simplified)
    - POST body includes:
      - model: "gpt-3.5-turbo"
      - messages: [
          {role: "system", content: "You are a witty data meme caption generator."},
          {role: "user", content: "Given the following data row: name=Mazda RX4, mpg=21, hp=110, wt=2.62, qsec=16.46. Provide a one-line meme caption in the style of a popular meme template."}
        ]
      - temperature: 0.7
      - max_tokens: 60
- Step 5: Apply API calls across all cars
  - Create a function get_caption(row, prompt_type) that builds the prompt from the row and calls the API
  - Map across all rows with purrr::pmap or map2, collecting results into a new column caption
  - Example approach (pseudocode):
    - cars <- tibble(name = rownames(mtcars), mpg = mtcars$mpg, hp = mtcars$hp, wt = mtcars$wt, qsec = mtcars$qsec)
    - prompts <- c(prompt_A, prompt_B)
    - captions <- map2_chr(1:nrow(cars), cars, ~ call_openai(.y, prompt = prompts[1]))
    - results <- cars %>% mutate(caption_prompt_A = captions)
- Step 6: Add error handling
  - Use tryCatch around API calls
  - Implement simple backoff if rate-limited (e.g., Sys.sleep(0.5) between calls)

Phase 3 — analysis of results (20–40 min)
- Step 7: Compute caption metrics
  - caption_len = nchar(caption)
  - word_count = str_count(caption, "\\S+")
  - Optional: sentiment_score using tidytext (bing lexicon) on caption words
- Step 8: Compare prompt variants
  - If you generated captions with both prompts, compute simple comparisons:
    - Distribution of caption_len by prompt
    - Proportion of captions containing a specific keyword (e.g., "mpg", "horsepower"/"hp")
- Step 9: Visualize
  - Histogram or density plot of caption lengths per prompt
  - Bar chart of average word count by prompt
  - Show a few representative captions as “meme samples” (text only)
- Step 10: Quick qualitative synthesis
  - Which prompt produced funnier captions? Which produced more data-insight style captions? Any surprising biases?

Phase 4 — reflection and extensions (15–25 min)
- Step 11: Reflect on prompt design and results
  - How did prompt wording influence tone, accuracy, and length?
  - What are the risks or biases in model outputs for meme captions?
  - Consider how to avoid unsafe or off-brand captions; discuss safety filters and moderation
- Step 12: Extensions (optional)
  - Add another dataset (e.g., built-in iris or a small CSV of student data) to test generalization
  - Use OpenAI’s moderation endpoint to flag unsafe content
  - Create a mini report that compares two models (gpt-3.5-turbo vs. gpt-4-turbo) in terms of caption quality and cost
  - Build a fun, shareable meme gallery by printing captions with car names in a ggplot facet plot

  - prompt_B <- "You are a witty data explainer meme generator. For the following car data, produce a one-line meme caption that highlights a data insight about mpg, hp, wt, and qsec, in a humorous way. Data: name={name}, mpg={mpg}, hp={hp}, wt={wt}, qsec={qsec}."
- API call function (phase 2)
  - Define function get_caption(row, prompt)
    - Build messages list:
      - system: "You are a witty data meme caption generator for graduate students."
      - user: glue::glue("Row data: name={row$name}, mpg={row$mpg}, hp={row$hp}, wt={row$wt}, qsec={row$qsec}. Provide a concise meme caption (max ~60 tokens).")
    - body <- list(model = "gpt-3.5-turbo", messages = messages, temperature = 0.7, max_tokens = 60)
    - res <- POST("https://api.openai.com/v1/chat/completions", add_headers(Authorization = paste("Bearer", Sys.getenv("OPENAI_API_KEY"))), content_type_json(), body = toJSON(body, auto_unbox = TRUE), encode = "json")
    - Check for errors; if ok, parse: content(res, as="parsed")$choices[[1]]$message$content
  - Apply prompts
    - captions_A <- pmap_chr(list(name=cars$name, mpg=cars$mpg, hp=cars$hp, wt=cars$wt, qsec=cars$qsec), ~ get_caption(list(name=..1, mpg=..2, hp=..3, wt=..4, qsec=..5), prompt_A))
    - captions_B <- pmap_chr(list(...), ~ get_caption(..., prompt_B))  // optional

  - results <- results %>% mutate(len_A = nchar(caption_A), words_A = str_count(caption_A, "\\S+"), len_B = nchar(caption_B), words_B = str_count(caption_B, "\\S+"))

    - sentiment <- tokens %>% inner_join(get_sentiments("bing"), by="word") %>% group_by(name) %>% summarise(sentiment_score = sum(ifelse(sentiment == "positive", 1, -1), na.rm = TRUE))
  - Visualize
    - ggplot(results, aes(x = factor(name), y = len_A)) + geom_col() + coord_flip() + labs(title="Caption length by car (Prompt A)")

- Reflection (phase 4)

  - Discuss potential biases, safety considerations, and cost implications
- Deliverables

Tips and caveats
- Start small: run the call for a few cars first to validate prompts and authentication, then scale up.
- Rate limits: respect OpenAI rate limits. If you hit 429s, implement a short backoff and retry strategy.
- Costs: be mindful of token usage. Use max_tokens modestly (30–60) for captions.
- Safety: consider using the moderation endpoint or a simple check to avoid unsafe or off-brand captions. Do not share or expose your API key.
- Reproducibility: save results to disk (write_csv) so you can re-run analyses or share the meme gallery.

Extensions for more rigor (optional)
- Compare model outputs using another model (e.g., gpt-4-turbo, if access allows) and quantify differences in length, sentiment, and humor proxies.
- Add a simple humor metric: count presence of humorous keywords, or crowdsource a quick score on several captions.
- Build a small Shiny app to display the meme captions in a gallery, with filters by prompt and by car feature ranges.

Example code sketch (short, to get started)
- Note: This is a compact scaffold. Adapt model, prompts, and error handling as needed.

- Data prep
  cars <- mtcars
  cars$name <- rownames(mtcars)
  cars <- cars %>% as_tibble() %>% select(name, mpg, hp, wt, qsec)

- API call function (simplified)
  get_caption <- function(row, prompt) {
    system_msg <- list(list(role = "system", content = "You are a witty data meme caption generator for graduate students."))
    user_msg <- list(role = "user", content = glue("Row data: name={row$name}, mpg={row$mpg}, hp={row$hp}, wt={row$wt}, qsec={row$qsec}. Provide a concise meme caption (max ~60 tokens)."))
    body <- list(model = "gpt-3.5-turbo", messages = c(system_msg, user_msg), temperature = 0.7, max_tokens = 60)
    res <- POST("https://api.openai.com/v1/chat/completions",
                add_headers(Authorization = paste("Bearer", Sys.getenv("OPENAI_API_KEY"))),
                content_type_json(),
                body = toJSON(body, auto_unbox = TRUE),
                encode = "json")
    if (http_error(res)) stop("OpenAI API error: ", status_code(res))
    content(res, as = "parsed")$choices[[1]]$message$content
  }

- Apply across cars (first 5 calls to test)
  promptA <- "You are a meme caption generator. For data row: name={name}, mpg={mpg}, hp={hp}, wt={wt}, qsec={qsec}. Create a one-line meme caption in the Doge style."
  captions_A <- map2_chr(cars$name, 1:nrow(cars), function(nm, i) {
    row <- cars[i, ]
    get_caption(row, promptA)
  })

- results
  results <- cars %>% mutate(caption_A = captions_A, len_A = nchar(caption_A), words_A = str_count(caption_A, "\\S+"))
