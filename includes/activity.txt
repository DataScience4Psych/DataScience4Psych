Meme-tastic data science activity: OpenAI API in the Tidyverse

Overview
This graduate-level activity lets you explore prompt design, API calls, response handling, and downstream analysis entirely in R using the tidyverse. You’ll issue a small set of meme-flavored prompts to OpenAI’s chat API, capture responses and usage metrics, and analyze results with tidyverse tools. The goal is to practice API integration, tidy data workflows, and thoughtful prompt engineering—while having fun with meme-friendly content.

Learning objectives
- Make OpenAI API calls from R (chat/completions endpoint) and handle responses safely.
- Integrate API results into tidyverse workflows (dplyr, purrr, tidyr, readr, stringr).
- Extract and summarize response metadata (content, token usage) and perform basic textual analytics (sentiment, length).
- Compare prompt designs using quantitative metrics and visualizations.
- Reflect on prompt engineering decisions and limitations of AI responses.

R script: core functions and data

  - library(jsonlite)

- Function to call OpenAI API (chat/completions)
  - This uses the model (e.g., gpt-4 or gpt-3.5-turbo), a system prompt to set persona, and a user prompt.

call_openai <- function(prompt,
                        model = "gpt-4",
                        system_text = "You are a witty, meme-loving data scientist who writes concise, actionable guidance with a dash of humor.",
                        temperature = 0.6) {
  req_body <- list(
    model = model,
    messages = list(
      list(role = "system", content = system_text),
      list(role = "user", content = prompt)
    ),
    temperature = temperature
  )
  res <- httr::POST(
    url = "https://api.openai.com/v1/chat/completions",
    httr::add_headers(Authorization = paste("Bearer", Sys.getenv("OPENAI_API_KEY"))),
    httr::content_type_json(),
    body = jsonlite::toJSON(req_body, auto_unbox = TRUE)
  )
  stop_for_status(res)
  raw <- httr::content(res, as = "text", encoding = "UTF-8")
  parsed <- jsonlite::fromJSON(raw, flatten = TRUE)
  reply <- parsed$choices[[1]]$message$content
  usage <- parsed$usage
  list(reply = reply, usage = usage)
}

- Optional: sentiment helper (tidytext-based)
  - We'll compute a simple sentiment score per reply using the Bing lexicon.

library(dplyr)
library(tidytext)

sentiment_score <- function(text) {
  docs <- tibble(doc = 1L, text = text)
  words <- docs %>% unnest_tokens(word, text)
  s <- words %>% inner_join(get_sentiments("bing"), by = "word") %>%
    count(sentiment) %>%
    pivot_wider(names_from = sentiment, values_from = n, values_fill = 0)
  pos <- s$positive %>% ifelse(is.null(.), 0, .)
  neg <- s$negative %>% ifelse(is.null(.), 0, .)
  pos - neg
}

2) Run the API for each prompt and collect results
- We'll issue each prompt, capture the reply and usage stats, and assemble a tidy data frame.

results <- map2_df(prompts$id, prompts$prompt, function(i, p) {
  out <- call_openai(p)
  tibble(
    id = i,
    prompt = p,
    reply = out$reply,
    prompt_tokens = out$usage$prompt_tokens,
    completion_tokens = out$usage$completion_tokens,
    total_tokens = out$usage$total_tokens
  )
})

3) Basic textual analytics on the responses
- Length metrics
results <- results %>%
  mutate(
    word_count = str_count(reply, "\\b\\w+\\b"),
    char_count = nchar(reply),
    # Very light sentiment score using tidytext
    sentiment = map_dbl(reply, sentiment_score)
  )

Tips and safety considerations
- Start with a small batch (e.g., 3 prompts) to test the workflow before scaling.
- Monitor usage and cost; set a temperature that balances creativity and reliability (e.g., 0.6–0.8).
- Do not send sensitive or proprietary data to the API. Treat prompts and responses as non-confidential.
- Store outputs locally if sharing results; consider adding an anonymized ID instead of raw prompts for reporting.

Extensions (for advanced students)
- Prompt engineering experiment: vary system_text and temperature across prompts, then compare response quality using predefined rubrics.
- Fine-grained sentiment: use a more robust sentiment model or lexicon (e.g., NRC, AFINN) and compare with Bing-based results.
- Code extraction: write a small parser to extract R code blocks from the replies and evaluate them (with caution, in a safe environment) for correctness and style.
- Prompt ensembles: combine multiple prompts per topic and aggregate outputs to reduce variance in responses.
- Reproducibility: wrap the workflow in an Rmarkdown/Literate Computing notebook; include the appendix with plots and a narrative.
